InkLink provides a **hosted** flow solution that offers the simplest integration path, with InkLink handling the entire request flow.

## Benefits

- **Zero Frontend Development** - No SDK or UI components needed
- **Minimal Integration Complexity** - Simple redirect-based flow
- **No SDK Overhead** - Keeps your application lightweight
- **Automatic Updates** - Always get the latest InkLink features

### Creating a Request

Request defines the structure and requirements of your data request flow. You can create requests through our UI builder or programmatically via API. In return, you will receive a `request_url` that you can share with users to request and verify their data.

## Creating Request via UI

Our visual Proof Request builder UI available in the InkLink WebApp provides an intuitive interface for creating request without code.

### Request Builder Sections

**1. Proof Request Configuration**

![Proof request configuration webpage](/images/request-1.png)

Define the core requirements of your verification:
| Field | Purpose | Impact on User Experience |
| ----- | ------- | ------------------------- |
|**Category**|Classify the request type|Displays matching icon and enables filtering|
|**Title**|Short, descriptive name|Shown in browser header and emails|
|**Description**|Clear instructions|Displayed before screenshot capture|
|**Example URL**|Reference link (optional)|Shows "See example" link to users|
|**User Validation**|Identity matching requirements|Determines acceptance criteria (see user validation models)|

**2. Landing Page Customization**

![Landing page design webpage](/images/request-2.png)

Configure the initial user experience:
| Setting | Description | Best Practice |
| ----- | ------- | ------------------------- |
|**Requester Name**|Your organization name|User your brand name for recognition|
|**Button Text**|CTA button label (max 25 chars)|Action-oriented text like "Start Verification"|
|**Notes**|Additional context|Keep brief - mention time required or benefits|

**2. Flow Configuration**

![Flow configuration webpage](/images/request-3.png)

Control the verification process:

**Animation Options:**

- `None` - Instant browser launch (fastest)
- `Short (2s)` - Brief loading indicator
- `Long (5s)` - Branded security message (recommended for first-time users)

**Capture Modes:**

- `Single` - One screenshot (highest completion rate)
- `Multiple - User Decides` - Flexible multi-page capture
- `Multiple Defined` - Structured wizard with required steps

**Post-Capture Features:**
| Feature | When Enabled | Use Case |
| ----- | ------- | ------------------------- |
|**Confirmation Modal**|Shows preview before submission|Reduces support tickets|
|**Allow Redaction**|Blur tool for sensitive data|Privacy compliance|
|**Include Screenshot**|Thumbnail on proof card|Quick visual verification|
|**Blockchain Anchoring**|On-chain proof hash|Immutable verification|

## Creating Request via API

For programmatic request creation, use the following endpoint:

```Typescript
POST https://app.inklink.com/api/request
```

**Minimal Request**
You can pass in optional fields to the request payload outlined in here {/* link to request page */}

```JSON
{
  "proof_request_description": "Verify account ownership"
}
```

**Response**

```JSON
{
  "request_id": "req_xxxx",
  "request_url": "https://app.inklink.com/request/req_xxxx"
}
```

## Tracking specific request session

If you need to track a particular user's request completion, you can use the following endpoint to create a unique `session_id` for tracking purposes.

```Typescript
POST https://app.inklink.com/api/request-session-id
```

**Request**

```bash
curl --location 'https://app.inklink.com/api/request-session-id' \\
--header 'x-api-key: secret_test_xxxx' \\
--header 'Content-Type: application/json' \\
--data '{
  "request_id": "req_xxx",
  "return_url": "https://your-app.com/request-complete?your-ref-id=xxx"
}'
```

**Response**

```JSON
{
  "session_id": "sid_xxx",
  "request_url": "https://app.inklink.com/request/req_xxx?session_id=sid_xxx"
}
```
